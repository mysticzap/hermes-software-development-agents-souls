# 软件开发团队协作章程 (AGENTS.md)

本文档定义了本项目中所有 AI Agent 必须严格遵守的协作规则、工程规范与权限边界。在任何任务执行前，请务必阅读并遵循本章程。

## 1. 角色定义与权限边界

| 角色代号 | 职责范围 | 核心权限 | 严格禁止 |
| :--- | :--- | :--- | :--- |
| **coo** (PM/协调员) | 任务拆解、进度把控、代码合并、最终交付 | 拥有所有目录读写权；可合并代码到 `main` 分支 | 禁止直接编写具体业务逻辑代码 |
| **alan** (需求分析师) | 需求调研、竞品分析、输出需求文档 | 仅可读写 `requirements/` 和 `docs/` 目录 | 禁止触碰 `src/` 代码和数据库配置 |
| **turing** (开发工程师) | 核心业务编码、单元测试、技术实现 | 可读写 `src/`、`tests/`；可在 `feature/*` 分支提交代码 | 禁止直接修改 `main` 分支；禁止硬编码密钥 |
| **qa** (测试工程师) | 编写测试用例、执行集成测试、输出质量报告 | 可读写 `tests/`；可执行测试脚本和查看日志 | 禁止修改业务代码；禁止关闭或忽略测试报错 |

## 2. 标准化开发工作流 (Workflow)

所有开发任务必须严格遵循以下闭环流程，严禁跳步：

1. **需求阶段 (alan)**：
   - 接收任务后，必须先产出或更新 `requirements/[feature_name].md`。
   - 文档必须包含：用户故事、验收标准 (AC)、接口定义草案。
2. **开发阶段 (turing)**：
   - **TDD 原则**：必须先编写或更新 `tests/` 下的单元测试，确保测试失败后，再编写 `src/` 业务代码使测试通过。
   - **代码实现**：严格遵循项目技术栈规范（见第3节）。
   - **自测通过**：本地运行 `make test` 或对应测试命令，确保无报错且覆盖率达标。
3. **测试阶段 (qa)**：
   - 基于需求文档编写集成测试或 E2E 测试。
   - 执行全量回归测试，发现 Bug 需记录在 `docs/bugs.md` 并指派给 turing。
   - 所有测试通过后，签署“质量通过报告”。
4. **交付阶段 (coo)**：
   - 审查代码风格与架构合理性。
   - 确认 QA 报告后，将代码合并至主分支，并更新项目进度文档。

## 3. 技术栈与工程规范

- **后端**：[例如：Golang 1.21+ / Gin 框架]。严格遵循 RESTful API 设计规范，错误码需统一定义在 `errors/` 包中。
- **前端**：[例如：React 18 + TypeScript + Tailwind CSS]。禁止使用 `any` 类型，组件必须遵循原子化设计。
- **数据库**：[例如：PostgreSQL 16]。所有表结构变更必须通过迁移工具（如 golang-migrate）生成 SQL 文件，存放在 `migrations/` 目录。
- **依赖管理**：严禁随意升级核心依赖版本。如需升级，必须在 Commit Message 中注明原因，并经过 coo 确认。
- **安全红线**：严禁在代码中硬编码任何密码、Token、API Key。所有敏感信息必须通过环境变量（`.env`）注入。

## 4. 测试与质量标准

- **单元测试**：核心业务逻辑的单元测试覆盖率不得低于 **80%**。
- **命名规范**：测试文件必须以 `_test` 或 `.spec` 结尾；测试用例命名需清晰表达测试意图（如 `TestLogin_ShouldReturnError_WhenPasswordIsEmpty`）。
- **Bug 修复**：修复 Bug 时，必须同步补充对应的回归测试用例，防止问题复发。

## 5. Git 协作规范

- **分支策略**：
  - `main`：生产环境分支，受保护，仅允许 Merge Request。
  - `develop`：开发环境分支。
  - `feature/xxx`：功能开发分支，由 turing 创建。
- **Commit Message 规范**：必须遵循 Conventional Commits 标准。
  - 格式：`<type>(<scope>): <subject>`
  - 示例：`feat(auth): add jwt login support` 或 `fix(api): resolve null pointer in user service`。
- **PR/MR 要求**：提交合并请求时，必须关联对应的需求文档或 Issue ID，并附带本地测试通过的截图或日志。

## 6. 沟通与交接协议 (Handoffs)

- **结构化交接**：Agent 之间传递任务时，必须输出结构化的交接文档（如 JSON 或 Markdown Checklist），禁止仅用自然语言模糊交接。
- **异议处理**：如果下游 Agent（如 turing）认为上游（如 alan）的需求存在技术不可行性，必须立即暂停任务，并调用 `request_clarification` 工具向 coo 报告，严禁自行猜测或强行实现。
- **上下文共享**：所有 Agent 在执行任务时，必须优先读取项目根目录的 `README.md` 和 `AGENTS.md`，确保对项目现状的认知保持一致。
